# 재구축 보완사항 상세 비교서 (As-Is vs To-Be)

## 1. 문서 목적
이 문서는 현재 프로젝트를 기준으로, 향후 실무 재구축 시 반드시 보완해야 할 사항을 상세 비교하여 실행 가능한 개선 계획으로 정리한다.

## 2. 비교 범위
- 현재 코드/구조 기준 파일
  - `scripts/sfe_sandbox.py`
  - `scripts/report_builder_v12.py`
  - `config/mapping.json`
  - `docs/workflow.md`
- 비교 축
  - 아키텍처, 데이터 계약, 조인/그레인, 품질, 운영, 배포, 보안, 거버넌스

## 3. 현재 상태 요약 (As-Is)
- 강점
  - Streamlit 기반 실행 콘솔과 HTML 보고서 생성 흐름이 이미 동작함
  - 매핑 파일(`config/mapping.json`)을 통한 컬럼 표준화 시도
  - 결과물 파일명 충돌회피 로직 존재
  - 도메인 통합(sales/targets/crm)과 보고서 출력 경험 축적
- 한계
  - 엔진/시나리오/템플릿 분리가 문서 수준 대비 코드에서 완전 분리되지 않음
  - 조인/그레인 검증이 체계적 게이트로 강제되지 않음
  - 설정(config) 스키마 검증, 변경 승인, 이력 추적 체계 미흡
  - 테스트 자동화 및 회귀 검증 부재
  - 실행 로그가 운영/감사 관점에서 구조화되어 있지 않음

## 4. 상세 보완사항 비교

### 4.1 아키텍처 분리
- As-Is
  - UI 코드와 처리 로직이 혼재되어 확장 시 결합도 증가 위험
  - 시나리오별 차이가 코드 분기/수동 조정으로 유입될 가능성 존재
- To-Be
  - 공통 엔진(`load -> mapping -> keys/join -> metrics -> payload -> render`)으로 모듈 분리
  - 시나리오 차이는 `config/scenarios/*.yaml` + `templates/*`로만 관리
- 보완 액션
  - `scripts/engine/*` 신설 및 책임 분리
  - `scripts/sfe_sandbox.py`는 호출/입출력 UI만 담당
- 우선순위: P0
- 완료 기준
  - 엔진 단독 실행 CLI 가능
  - 동일 엔진으로 monthly/quarterly/business 모두 실행

### 4.2 데이터 계약(Data Contract)
- As-Is
  - 표준 컬럼 개념은 있으나 강제 스키마와 타입 검증이 약함
  - 한글/영문 혼합 컬럼과 포맷 불일치 발생 가능
- To-Be
  - 표준 컬럼 사전 + 필수 ID + 타입 + 시간키(`metric_month=YYYY-MM`) 계약 고정
- 보완 액션
  - `config/schema/master_columns.yaml` 작성
  - 로딩 직후 schema validator 단계 추가
- 우선순위: P0
- 완료 기준
  - 필수 컬럼 누락/타입 오류 시 실행 즉시 실패
  - 스키마 위반 리포트 자동 생성

### 4.3 매핑 정책 및 하위호환
- As-Is
  - `mapping.json` 사용은 있으나 승인/리뷰/이력 관리 프로세스가 약함
  - 별칭 확장 시 기존 리포트 영향 분석이 수동
- To-Be
  - 매핑 변경은 정책 기반 승인 + 변경 이력 필수
  - 기존 표준키 하위호환 강제
- 보완 액션
  - `mapping_change_log.md` 또는 JSON changelog 도입
  - 매핑 변경 시 샘플 회귀 테스트 자동 실행
- 우선순위: P1
- 완료 기준
  - 매핑 변경마다 영향 시나리오 목록 자동 출력

### 4.4 테리토리 매핑 정책(신규 필요)
- As-Is
  - 현재 맵 빌더(`scripts/map_data_builder.py` + `templates/spatial_preview_template.html`)는 `rep/month/date/hospital/lat/lon` 중심 페이로드로 동작
  - 테리토리 전용 키(`territory_id`, `territory_name`) 매핑은 아직 구현/적용되어 있지 않음
  - 지점/Region 수준 일반 매핑은 존재하나 테리토리 코드/명 불일치 시 우선순위 및 예외 처리 기준이 문서화되어 있지 않음
- To-Be
  - 테리토리 표준키(`territory_id`, `territory_name`, `branch_id`, `branch_name`)를 신규 데이터 계약으로 정의
  - 매핑 우선순위(코드 우선, 이름 보조)와 충돌 처리(1:N, N:1, 미매핑) 규칙을 강제
- 보완 액션
  - `config/mapping.json`에 테리토리 별칭 및 하위호환 alias 명시
  - `config/territory_mapping_rules.yaml` 신설(우선순위, 예외코드, 강제값 규칙)
  - 조인 전 `validate_territory_mapping` 단계 추가(미매핑률/충돌률 임계치)
  - 테리토리 변경 이력(`territory_mapping_change_log.md`)과 영향 리포트 자동 생성
- 우선순위: P0
- 완료 기준
  - 입력 소스별 테리토리 미매핑률 1% 이하
  - 테리토리 충돌 케이스 자동 탐지 및 실행 실패(Fail-Fast)
  - 월별 테리토리 매핑 변경 내역과 영향 대상 리포트 자동 출력

### 4.5 조인/그레인 정책
- As-Is
  - ID 조인 원칙은 있으나 N:N/행 폭증 방지 게이트가 코드 전역에 일관 적용되지 않음
  - Long/Wide 변환 타이밍이 산발적일 가능성
- To-Be
  - 마스터 grain 명시 후, 모든 입력을 동일 grain으로 정규화 후 조인
  - Fail-Fast 임계치(null/중복/행 증가율) 강제
- 보완 액션
  - 조인 전 검증 함수 공통화 (`validate_keys`, `validate_join_delta`)
  - 임계치 시나리오별 config 주입
- 우선순위: P0
- 완료 기준
  - 조인 실패 시 단계/원인/샘플키 로그 저장
  - 행 폭증 케이스 자동 차단

### 4.6 지표 계산 표준화
- As-Is
  - 고급 지표 로직(HIR/RTR/BCR/PHR 등)은 존재하나 입력 품질 의존도가 높고 추적성이 약함
- To-Be
  - 지표 정의서 + 계산식 버전 관리 + 입력 품질 검증 연결
- 보완 액션
  - `metrics_catalog.yaml`로 지표 메타(정의/식/단위/허용 범위) 관리
  - 지표별 sanity check 자동화
- 우선순위: P1
- 완료 기준
  - 지표별 결측률/범위 위반 자동 리포트

### 4.7 테스트 및 품질보증
- As-Is
  - 수동 확인 중심, 회귀 테스트 체계 부족
- To-Be
  - 단위 테스트 + 통합 테스트 + 리그레션 데이터셋 기반 검증
- 보완 액션
  - `tests/`에 stage별 fixture 도입
  - 시나리오별 골든 결과 스냅샷 비교
- 우선순위: P0
- 완료 기준
  - CI에서 최소 테스트 통과 없이는 병합 금지

### 4.8 실행 로그/관측성(Observability)
- As-Is
  - 실행 결과 확인은 가능하나 운영 관점 메타데이터 표준화 부족
- To-Be
  - run_id 단위로 단계별 상태/건수/시간/오류를 구조화 저장
- 보완 액션
  - `output/logs/{run_id}.json` 표준 포맷 도입
  - 실패 유형 분류 코드(LOAD_FAIL, SCHEMA_FAIL, JOIN_FAIL 등) 적용
- 우선순위: P0
- 완료 기준
  - 실패 원인 재현에 필요한 정보가 로그만으로 충분

### 4.9 배포/릴리즈 관리
- As-Is
  - 결과물 생성은 되지만 변경 릴리즈 기준과 롤백 절차가 약함
- To-Be
  - 릴리즈 단위 변경관리(영향 범위, 검증 결과, 롤백 포인트) 정례화
- 보완 액션
  - `release_change_log` 운영
  - 배포 전 체크리스트 자동화 스크립트 추가
- 우선순위: P1
- 완료 기준
  - 배포 건마다 변경 이력과 검증 증적 보관

### 4.10 보안/거버넌스
- As-Is
  - 데이터 보존 원칙은 존재하나 권한/감사/민감정보 처리 절차가 문서화 미흡
- To-Be
  - 역할 기반 권한, 감사추적, 민감정보 최소 노출 원칙 적용
- 보완 액션
  - 실행자/승인자 분리 정책 명시
  - 로그/리포트 민감정보 마스킹 규칙 적용
- 우선순위: P1
- 완료 기준
  - 감사 요청 시 실행/변경/배포 이력 즉시 제출 가능

### 4.11 인코딩/국문 데이터 안정성
- As-Is
  - 환경에 따라 한글 깨짐 발생 가능성 존재
- To-Be
  - UTF-8 강제 및 검증 루틴 표준화
- 보완 액션
  - 실행 전 환경 검증(`PYTHONIOENCODING=utf-8`) 체크
  - 깨진 콘솔 출력 기반 수정 금지 규칙 자동 점검
- 우선순위: P0
- 완료 기준
  - 국문 컬럼/값이 모든 단계에서 무손실 유지

## 5. 우선순위 로드맵

### Phase 0 (1주): 안정화 최소선
- 데이터 계약 강제
- 조인 Fail-Fast 게이트 구현
- 구조화 로그(run_id) 도입

### Phase 1 (2주): 구조 분리
- 엔진 모듈 분리
- 시나리오 config 템플릿화
- 템플릿 렌더 계층 분리

### Phase 2 (2주): 품질 자동화
- 단위/통합/회귀 테스트 구축
- 지표 sanity check 자동화
- 매핑 변경 영향 분석 자동화

### Phase 3 (지속): 운영 성숙화
- 릴리즈/롤백 운영
- 거버넌스/감사 고도화
- KPI 모니터링 운영

## 6. KPI 및 목표값(권장)
- 신규 시나리오 리드타임: 3일 이내
- 수동 수정 없는 성공 실행률: 95% 이상
- 조인 오류 재발률: 월 5% 이하
- 매핑 재사용률: 80% 이상
- 평균 리포트 생성 시간: 10분 이내

## 7. 즉시 실행 가능한 액션 (다음 스프린트)
1. `config/schema` 추가 및 validator 구현
2. 조인 검증 공통 함수 도입
3. run_id 기반 JSON 로그 표준화
4. 시나리오 YAML 3종(monthly/quarterly/business) 초안 적용
5. 회귀 테스트용 샘플 데이터셋 1세트 확정

## 8. 결론
현재 프로젝트는 실무 경험이 반영된 강한 기반을 갖고 있으나, 재구축 관점에서는 "운영 강제력"과 "자동 검증 체계"가 핵심 보완 지점이다.
재구축의 성공 조건은 코드 추가보다도, 정책을 코드로 강제하는 구조(스키마 검증, Fail-Fast, 테스트, 구조화 로그)에 있다.
